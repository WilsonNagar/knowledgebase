# Customizing SystemUI (OEM customizations)

## Overview

Customizing SystemUI (OEM customizations) refers to the process of modifying SystemUI to meet OEM (Original Equipment Manufacturer) requirements, including visual customizations, functional modifications, theme support, and brand-specific features. OEM customizations allow manufacturers to differentiate their devices through custom SystemUI appearance, additional features, and brand-specific UI elements. Understanding how to customize SystemUI for OEM requirements is essential for AOSP development, as it explains how to modify SystemUI components, how to use resource overlays, how to implement custom features, how to apply themes, and how to maintain compatibility with AOSP updates. This guide provides a comprehensive overview of customizing SystemUI for OEM requirements, customization methods, resource overlays, code modifications, theme support, and best practices.

Think of customizing SystemUI like customizing a car's interior: just as a car manufacturer can customize the interior (custom colors, materials, features, brand-specific elements) while maintaining the core functionality (engine, transmission, safety systems), OEMs can customize SystemUI (custom appearance, features, brand elements) while maintaining core Android functionality (framework, services, APIs).

### Customization Methods

#### Resource Overlays

**Resource Overlays:**
- Override AOSP resources
- Custom layouts
- Custom drawables
- Custom colors
- Custom strings

**Overlay Structure:**
```
vendor/overlay/SystemUI/
├── res/
│   ├── layout/
│   ├── drawable/
│   ├── values/
│   └── ...
└── AndroidManifest.xml
```

**Overlay Example:**
```xml
<!-- AndroidManifest.xml -->
<overlay
    android:targetPackage="com.android.systemui"
    android:priority="1000" />

<!-- res/values/colors.xml -->
<resources>
    <color name="status_bar_background">#FF0000</color>
</resources>
```

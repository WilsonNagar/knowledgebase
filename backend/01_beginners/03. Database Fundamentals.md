# Database Fundamentals

## Overview

Databases are essential for storing and managing data in backend systems. This guide covers relational (SQL) and non-relational (NoSQL) databases, ACID properties, database design principles, normalization, indexing, transactions, and when to choose SQL vs NoSQL.

## Deep Explanation

### SQL vs NoSQL

**SQL (Relational)**:
- Structured data with schemas
- ACID compliance
- Relationships between tables
- Examples: PostgreSQL, MySQL, SQL Server

**NoSQL (Non-Relational)**:
- Flexible schemas
- Horizontal scaling
- Various data models
- Examples: MongoDB, Cassandra, Redis

### ACID Properties

**Atomicity**: All operations succeed or all fail
**Consistency**: Data remains valid
**Isolation**: Concurrent transactions don't interfere
**Durability**: Committed data persists

### Database Design

**Normalization**:
- Eliminate redundancy
- Reduce data duplication
- Improve data integrity

**Denormalization**:
- Improve read performance
- Accept some redundancy
- Common in NoSQL

### Indexing

**Purpose**: Speed up queries
**Trade-off**: Faster reads, slower writes, more storage

**Common Indexes**:
- Primary key (automatic)
- Foreign key
- Frequently queried columns

## Real Code Examples

### Example: User and Posts Schema

**SQL Schema**:
```sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE posts (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES users(id),
    title VARCHAR(200) NOT NULL,
    content TEXT,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_posts_user_id ON posts(user_id);
```

**NoSQL Schema (MongoDB)**:
```json
{
  "_id": ObjectId("..."),
  "name": "John Doe",
  "email": "john@example.com",
  "posts": [
    {
      "title": "Post 1",
      "content": "Content..."
    }
  ]
}
```

## Hard Use-Case: Choosing Database Type

### Problem Statement

Decide between SQL and NoSQL for an e-commerce platform.

### Solution

**Use SQL (PostgreSQL) for**:
- User accounts (ACID important)
- Orders and payments (transactions critical)
- Inventory management (consistency required)

**Use NoSQL (MongoDB) for**:
- Product catalogs (flexible schema)
- User sessions (fast reads)
- Analytics data (write-heavy)

**Hybrid Approach**:
- SQL for transactional data
- NoSQL for high-volume, flexible data
- Cache layer (Redis) for frequently accessed data

## Edge Cases and Pitfalls

### Common Mistakes

1. **Over-normalization**
   - Too many joins hurt performance
   - Balance normalization with performance

2. **Missing Indexes**
   - Slow queries on large tables
   - Index foreign keys and frequently queried columns

3. **N+1 Query Problem**
   - Querying in loops
   - Use joins or batch loading

## Quiz

### Question 1
What does ACID stand for?

**A)** Atomicity, Consistency, Isolation, Durability  
**B)** Accuracy, Consistency, Integrity, Durability  
**C)** Atomicity, Correctness, Isolation, Durability  
**D)** Availability, Consistency, Isolation, Durability

**Answer: A** - ACID stands for Atomicity, Consistency, Isolation, Durability.

### Question 2
When should you choose NoSQL over SQL?

**A)** When you need ACID compliance  
**B)** When you need flexible schemas and horizontal scaling  
**C)** When you need complex joins  
**D)** When you need strict data consistency

**Answer: B** - NoSQL is better for flexible schemas and horizontal scaling.
